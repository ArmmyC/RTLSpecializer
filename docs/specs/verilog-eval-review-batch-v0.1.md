# Feature Spec: VerilogEval Review Batch v0.1

## 1. Goal

Create the first real human-review data workflow for RTLSpecializer using locally supplied VerilogEval-style public data.

The current repository has strong infrastructure: schemas, validators, draft import, review promotion, release assembly, leakage checks, and deterministic evaluation. However, the current handwritten seed rows were generated by repo scripts and should be treated as synthetic seed/smoke data, not as independently human-reviewed RTL training data.

This feature has two connected goals:

1. Prepare a small VerilogEval-derived review batch, targeting 10-20 examples that a human reviewer can inspect and edit.
2. Clean up seed-row provenance language so the existing synthetic seed rows are not confused with independently reviewed training data.

The workflow should be:

```text
local VerilogEval data or local VerilogEval manifest
  -> conservative dataset_v0.1 draft rows
  -> ranked/filtered review batch of 10-20 rows
  -> review packet
  -> human edits reviewed_rows.jsonl
  -> promotion
  -> release/evaluation smoke flow
```

Codex/scripts may convert and rank data, but must not mark rows as human-reviewed. The user or another human reviewer must make the final review decision.

## 2. Non-goals

Do not build:

- Model training.
- Model inference.
- External LLM calls.
- Automatic VerilogEval downloads.
- Web scraping.
- EDA execution.
- RTL simulation, synthesis, equivalence, toggle, or power analysis.
- Automatic promotion to `validated` without human edits.
- License approval automation.
- New dataset schema versions.
- Private/company RTL ingestion.

## 3. Background and rationale

VerilogEval is a small public benchmark of Verilog tasks derived from HDLBits. It is useful as a first public-data source because it is much smaller and easier to inspect than large scraped RTL corpora.

For RTLSpecializer, we should not use VerilogEval as pure code-generation training data yet. Instead, we should convert selected examples into review-style tasks:

- explain RTL behavior,
- identify possible correctness risk,
- discuss cycle/reset/interface risk,
- avoid unsupported area/activity/power claims,
- propose verification plan,
- produce conservative review answers.

The first target is not scale. The first target is a small, auditable, manually reviewed batch.

## 4. Assumptions

- The user manually obtains or stages VerilogEval data locally.
- Scripts read only local files.
- The existing `verilog_eval` adapter currently delegates to the manifest adapter when `manifest.jsonl` is present.
- `dataset_v0.1`, `rtl_task_v0.1`, and `rtl_answer_v0.1` remain unchanged.
- Draft rows must remain `review_status: draft`.
- Human-reviewed rows flow through the existing review/promotion workflow.
- Python standard library only.

## 5. User stories

- As a human reviewer, I want 10-20 clear VerilogEval-derived examples, so I can review them without being overwhelmed.
- As a dataset maintainer, I want imported rows to stay draft until a human edits/reviews them.
- As a project lead, I want seed rows to be labeled honestly, so we do not confuse synthetic smoke data with human-reviewed training data.
- As a future trainer, I want VerilogEval rows to include provenance/license notes before they enter a release.
- As an evaluator, I want the reviewed batch to pass release and evaluation smoke tests before it is treated as training data.

## 6. UX / UI requirements

No graphical UI.

Add a local review-batch CLI:

```bash
python scripts/dataset/prepare_verilog_eval_review_batch.py \
  --input data/raw_public/verilog_eval/manifest.jsonl \
  --output-dir data/review/verilog_eval_batch_001 \
  --limit 20 \
  --json
```

Supported options:

```text
--input <path>                 local VerilogEval manifest or supported local JSONL
--output-dir <dir>
--limit <int>                  default 20
--source <source enum>         default public_verilog_eval
--license <license text>
--prefer-task-type <task type> optional repeatable
--json
```

Output directory:

```text
data/review/verilog_eval_batch_001/
  draft_rows.jsonl
  selected_rows.jsonl
  review_packet/
    README.md
    review_manifest.jsonl
    rows/<id>.json
    rows/<id>.review.md
  selection_report.json
```

Text output example:

```text
VerilogEval review batch prepared.

Discovered rows: 156
Draft rows: 156
Selected rows: 20
Rejected rows: 0
Output: data/review/verilog_eval_batch_001
Next: edit reviewed_rows.jsonl, then run promote_reviewed_rows.py
```

JSON output must include `ok`, discovered/imported/selected/rejected counts, output paths, warnings, and errors.

## 7. Functional requirements

### FR-1: Keep all source data local

Scripts must not download VerilogEval, scrape websites, call GitHub APIs, or call external services. They should only read files provided through `--input`.

### FR-2: Accept manifest-first VerilogEval workflow

The easiest path must work with the existing public manifest format.

If `--input` is a `manifest.jsonl`, reuse existing import code or adapter behavior to create draft rows.

The manifest rows should map VerilogEval examples into artifacts such as:

```text
rtl_code
before_rtl_code
after_rtl_code
testbench
lint_log
synthesis_report
toggle_report
```

For VerilogEval, the most common useful fields are expected to be:

- problem prompt/specification,
- reference/canonical RTL solution,
- optional testbench/checker,
- task ID/name/category metadata.

### FR-3: Optionally accept simple VerilogEval JSONL exports

If practical, support a conservative JSONL parser for local VerilogEval-style exports.

Accepted field aliases may include:

```text
task_id, problem_id, id, name
prompt, instruction, description, spec
canonical_solution, solution, rtl, verilog, reference_solution
test, testbench, checker
category, difficulty, source
```

If the parser cannot confidently identify prompt/spec and RTL, reject the example with a clear error.

This parser is optional if it becomes risky. Manifest-first support is required.

### FR-4: Generate conservative draft rows only

All imported VerilogEval rows must be:

```text
source: public_verilog_eval
review_status: draft
split: unsplit
created_by: script
```

Draft assistant answers must remain conservative and must not claim:

- verified correctness,
- area improvement,
- activity improvement,
- power improvement,
- successful simulation,
- successful synthesis.

### FR-5: Select a reviewable subset

Select up to `--limit` examples for human review.

Selection should prefer examples that are useful for RTL review:

- sequential logic,
- counters,
- FSMs,
- shift registers,
- mux/decoder/control logic,
- examples with non-trivial signal names,
- examples with both prompt/spec and RTL.

Avoid or rank lower:

- extremely tiny one-line combinational gates,
- examples with missing RTL/spec,
- ambiguous or empty metadata,
- examples with huge artifacts,
- examples that fail validation.

Selection must be deterministic.

### FR-6: Create review packet

Use or wrap the existing review-packet generator to create Markdown files for selected rows.

Each review Markdown file must show:

- row ID,
- source/provenance/license,
- task type,
- design family,
- prompt/spec artifact if available,
- RTL artifact,
- extracted RTL summary,
- draft answer,
- review checklist.

The checklist must explicitly ask the reviewer to confirm:

- useful task type,
- answer grounded in real signals/code,
- no unsupported verification claim,
- no unsupported area/activity/power claim,
- reasonable cycle/reset/interface reasoning,
- reasonable verification plan,
- license/provenance acceptable for project use.

### FR-7: Produce editable reviewed_rows template

Create an editable file:

```text
data/review/verilog_eval_batch_001/reviewed_rows.jsonl
```

This file should initially contain the selected draft rows or a clearly marked copy for human editing.

Important: these rows must still be `review_status: draft` until promotion or until the human edits them and promotion changes them. Do not silently mark them `validated`.

### FR-8: Improve seed-row provenance honesty

The current generated seed rows should not be confused with independently human-reviewed rows.

Implement a minimal, non-breaking cleanup:

- update docs to call `data/golden/golden_v0.1.jsonl` synthetic seed/smoke rows unless they receive real human review,
- update dataset guidelines to state that `review_status: reviewed` in generated seed data means script-generated seed acceptance, not named engineer review,
- add a TODO or issue-style note in docs recommending future explicit reviewer metadata,
- do not break existing tests that expect `data/golden/golden_v0.1.jsonl` to exist.

Do not rename files in this spec unless all tests/docs are updated safely.

### FR-9: Add reviewer metadata recommendation

Add documentation for future reviewed rows:

```json
"review_metadata": {
  "reviewed_by": "<name_or_role>",
  "reviewed_at": "YYYY-MM-DD",
  "review_method": "manual_rtl_review",
  "review_notes": "..."
}
```

Since this may not be part of the current schema, document it as a recommended future schema field or provenance note, not as a required validator field.

### FR-10: Add tests

Add tests under:

```text
tests/dataset/test_verilog_eval_review_batch.py
```

Required tests:

- prepares a review batch from a local manifest fixture,
- selected rows are draft and unsplit,
- selected rows use `source: public_verilog_eval`,
- review packet files are created,
- reviewed_rows.jsonl template is created,
- selection is deterministic,
- invalid/missing artifacts are rejected clearly,
- `--limit` is respected,
- CLI `--json` output is parseable,
- seed provenance docs mention synthetic seed/smoke status.

### FR-11: Add fixtures

Add a tiny local fixture:

```text
tests/fixtures/verilog_eval_review/manifest.jsonl
```

Include 3-5 small synthetic public-safe examples shaped like VerilogEval-style rows. Do not copy large real benchmark content into tests.

### FR-12: Add docs

Create:

```text
docs/dataset/verilog_eval_review_workflow.md
```

Update:

```text
docs/dataset/dataset_guidelines.md
README.md
```

Docs must explain:

- how to stage VerilogEval data locally,
- how to prepare a review batch,
- how the human review step works,
- how to promote reviewed rows,
- how to build a release with reviewed VerilogEval rows,
- how to run baseline evaluation,
- why generated seed rows are not enough for training,
- why VerilogEval rows remain drafts until human review.

## 8. Technical requirements

### 8.1 Architecture

Prefer composing existing tools:

```text
prepare_verilog_eval_review_batch.py
  -> import_public_dataset / VerilogEvalAdapter / ManifestAdapter
  -> prepare_review_packet.py helpers if reusable
  -> validation helpers
```

Avoid duplicating row generation logic when `build_draft_row` can be reused.

### 8.2 Standard library only

No new third-party dependencies.

### 8.3 Security

- Treat VerilogEval/local files as untrusted data.
- Read files as text only.
- Do not execute RTL.
- Do not execute testbenches.
- Do not call simulators.
- Do not call synthesis tools.
- Do not download data.
- Do not shell out.
- Do not include private/company RTL.

## 10. Data lifecycle

The intended first real data lifecycle is:

```bash
python scripts/dataset/prepare_verilog_eval_review_batch.py \
  --input data/raw_public/verilog_eval/manifest.jsonl \
  --output-dir data/review/verilog_eval_batch_001 \
  --limit 10 \
  --license "<verified license/provenance note>" \
  --json

# Human edits data/review/verilog_eval_batch_001/reviewed_rows.jsonl

python scripts/dataset/promote_reviewed_rows.py \
  --input data/review/verilog_eval_batch_001/reviewed_rows.jsonl \
  --output data/processed/verilog_eval_validated_v0.1.jsonl \
  --report data/reports/verilog_eval_validated_v0.1_report.json \
  --json

python scripts/dataset/build_dataset_release.py \
  --release-name release_v0.1_plus_verilog_eval_001 \
  --input data/golden/golden_v0.1.jsonl \
  --input data/processed/verilog_eval_validated_v0.1.jsonl \
  --output-dir data/releases \
  --seed 7 \
  --allow-source-overlap \
  --json

python scripts/eval/make_baseline_candidates.py \
  --dataset data/releases/release_v0.1_plus_verilog_eval_001/test.jsonl \
  --output data/eval/candidates/rule_baseline_verilog_eval_001.jsonl \
  --json

python scripts/eval/evaluate_answers.py \
  --dataset data/releases/release_v0.1_plus_verilog_eval_001/test.jsonl \
  --candidates data/eval/candidates/rule_baseline_verilog_eval_001.jsonl \
  --output-dir data/eval/runs/rule_baseline_verilog_eval_001 \
  --json
```

## 11. Edge cases

Handle:

- input path missing,
- input manifest malformed,
- unsupported local VerilogEval JSONL shape,
- duplicate source IDs,
- missing license option,
- examples with no RTL artifact,
- examples with no prompt/spec,
- examples with very large artifacts,
- zero selected rows,
- selected rows fail validation,
- output directory already exists,
- Unicode in prompt or RTL comments.

## 13. Definition of done

Done only when:

- A local VerilogEval review batch can be prepared without downloads.
- Selected rows are valid `dataset_v0.1` draft rows.
- Review packet files are generated.
- `reviewed_rows.jsonl` template exists for human editing.
- Seed/golden docs honestly explain synthetic seed/smoke status.
- Tests cover manifest fixture, deterministic selection, limit handling, draft status, source enum, packet creation, CLI JSON, and docs.
- No row is marked `validated` without human review/promotion.
- No model training, LLM calls, EDA execution, downloads, or schema changes are introduced.
